Design Evolution

From Alpha Prototype

For the alpha prototype, we introduced some major design changes to our system. One of the major changes was the move from gestural inputs to button inputs. This provided the user more freedom and gave visibility, removing the user’s reliance on their memory. We also made the application compatible with Talkback, which was a major usability criteria when it comes to the visually impaired community.

For our beta prototype, the design changes we implemented are more subtle but, we believe, improve the user experience, simplify the application and improve our score on Nielsen’s usability criteria. These modifications are based on feedback we obtained from fellow classmates as well as changes we felt were important.

Removal of Main Menu page

In our alpha prototype, our design had 3 menus as shown in the images below.

Explained simply, the alpha version starts in the main menu page where the user can select to enter either the intersection mapping menu or the light detection menu; and the application flows from there (This is further explained in the alpha prototype user manual). However, there is actually no utility provided by this page.

For this reason, we have “stripped” the application from this page and greatly simplified the application in other menus, as shown below.

Now, when the application starts, the intersect-assist page shows up showing two buttons. When the user chooses the intersection mapping button, the intersection information is provided via audio, while the application page doesn’t change. When the user chooses the light detection button, the second page starts where a camera detects the light color. When the user completes crossing the road, they are required to click the I’m done button, which takes them back to the original page. This modification relates to Nielsen’s Aesthetics and Minimalist design heuristic.

Providing the mapping information in a spatial format

Our second modification is in the way we provide the mapping information to the user. Initially, when the user prompts the application for the intersection information, the following example message would be outputted:

“Intersection: St-Catherine and Guy, 4-way configuration. St-Catherine is a one-way street, three lanes. Traffic flows in the opposite direction you are facing. Guy is a two-way street, four lanes. The closest two lanes ahead flow from your left to your right. The further two lanes flow from your right to your left.”

The problem with this structure is in the wording used to provide the spatial information. Additionally, some of the information provided wasn’t found essential by the users during our low-fidelity prototype testing. However, at the time, we thought that keeping it couldn't hurt. However, we have now decided to remove some of that information in order to speed up the delivery of the information. The new message is as follows:

Audio from both ears to simulate information is spatially in front of the user.
“Intersection: St-Catherine and Guy, 4-way configuration.”
Audio from the right ear only, indicating to the user that this relates to information from their right.
“St-Catherine is a one-way street, three lanes.”
Audio from both ears to simulate the information is spatially in front of the user.
“Guy is a two-way street, four lanes.”

This modification relates to Nielsen’s Match between system and the real world heuristic, since the user can spatially understand how the information relates to the real world without actually having to say where the street is.

Addition of feedback when awaiting light status

The last modification we implemented in our beta prototype is in the light detection part of the application. In the alpha phase of the application, when the user clicks the light detection section, the application doesn’t inform the user of what it’s doing. We felt that this would cause the user confusion and would not be sure how to proceed. For this reason, we implemented a message that plays every 5 seconds to the user which says:

Waiting for light to turn green

This modification relates to Nielsen’s Visibility of system status heuristic in that the user will know the status of the application throughout the use of the system.

Prototype Implementation

The overall architecture of our implementation consist on a smartphone (Android-based) application and a server side component that has the responsibility of doing all the heavy processing related to intersection mapping calculation and traffic light detection. Due to some estimations on the complexity of the feature and the most needed accuracy of an implementation to have a potential value for our users, we decided to remove the veering detection feature from our Beta Prototype. Hence, our Beta Prototype now includes the traffic light detection feature, which was missing from our ALpha release. This feature has brought the following changes to our system archiecture:

The traffic light detection feature heavily relies of the implementation provided by the Yolo Darknet reserach project as an starting point for our light detection mechanism. This real-time object detection system provides us with the capability of detection traffic light in daily scenarios. On top of the results provided by this implementation we have created an specific routine to detect the status of the traffic light. At this moment, we do not considered that this feature has been properly tested in enough scenarios to consider it safe for our end users. We expect that in a near future we will be able to run enough test scenarios to bring this application to our potential users.
