User and Installation Manual

User Introduction

The intersect-Assist is an application built to enrich the abilities of members of the visually impaired community. Specifically, it’s built with the purpose of assisting visually impaired users swith three tasks involving intersection crossing: Mapping information regarding intersections. Traffic Light detection. Veering assistance. It is very important that users still rely on their own skills in navigating around intersections. This application will do its best to assist in accomplishing those task but shouldn’t be solely relied on. This is a preliminary prototype of the proposed system intended to showcase the possibilities with such an application. It is also meant as a test of usability and the feedback obtained will help improve this system and increase the chances of its success.

Design Evolution

The following four sections of sketches will propose different functionalities across the three tasks that Intersect-Assist will perform and shows possible versions of how the user will interact with the smartphone.

Design Change #1: Terminology

The terminology used to describe intersections wasn’t clear. For example, using the words “Traffic flows in the direction you are facing…” proved confusing to both participants. A more suitable explanation is “Traffic flows from back to front…” For this prototype, we implemented it with explanations that are more intuitive to the users.

Design Change #2: Gestures within the Application

The gestures original described were “Swipe Upwards”, “Swipe Downwards” and “Tap once”. We intended for swiping upwards or downwards to be used in a spatial context, meaning no matter which way you put the phone, the user should swipe towards the sky for upwards and ground for downwards. However, we didn’t pay attention to how users wear the mobile device to use this and other applications, nor did we know what the users are used to doing when an application requires them to swipe upwards. Since they wear the smartphones upside down around their necks, they hold the phone horizontally and swipe in the direction relative to the phone. These adjustments are reflected in this version.

Design Change #3: Portrayal of Information and Interaction

Our design has gone through several iterations in the low fidelity phase. In our low-fidelity testing, we provided users with three different experiences:

  1. Mapping and veering provided with partially spatialized audio.
  2. Mapping provided with regular audio and veering was provided with partially spatialized audio.
  3. Mapping provided with regular audio and veering provided with haptic feedback bracelets.

After performing the testing, the favoured prototype was #2 above. Although the users didn’t feel a substantial difference between haptics and stereo for veering, from the point of view of feasibility, we decided to go with audio beeping. Additionally, users felt that we can enrich the experience of beeping with the addition of some form of speech describing how much veering is occurring. We decided to implement it via a clock analogy, i.e. if the user is veering towards 10 o’clock, the system will tell them you are moving towards 10 o’clock. turn towards 12 o’clock. This would be overlaid with the beeping from ear to ear.

Usability Goals

The users you should evaluate the prototype with must be from the visually impaired community, preferably the INLB community. We described two types of users in our project proposal: the blind user and the low-vision user. For the mapping side of the application, both personas would greatly benefit from the proposed system. However, veering is encountered to a much greater degree by blind users than low vision users. For this purpose, we would prefer testing be done with more blind users than low vision users. But it is still very important that the latter persona still be tested with to ensure the application has a sense of generality for both user groups.

Goal # 1

The user shouldn’t need to play the introduction to the application more than two times, which includes the information on allowances.

PASS/FAIL: If the user needed to repeat the introduction more than two times, the system fails to meet this goal.

Goal # 2

The described allowances below shouldn’t cause any confusion to the user.

  • Swipe up, meaning relative to the screen.
  • Swipe down, meaning relative to the screen.
  • Double tap, meaning to double tap the screen.

PASS/FAIL: If, for example, the user swiped upwards when she intended to swipe downwards(or any variant of this example), and it wasn’t due to any accidental swiping, the system fails to meet this goal.

Goal # 3

The rendering of information regarding mapping and/or veering should be coherent with what the user intends to accomplish.

PASS/FAIL: If the user is confused and requires the help of someone else, the system fails to meet this goal.

Goal # 4

The user should be capable of skipping any information she doesn’t want to hear.

PASS/FAIL: If the user is not capable of skipping certain pieces of information, the system fails to meet this goal.

Goal # 5

The user should at all time know what input is expected of her in order to enter the mode he requires.

PASS/FAIL: If, at any time, the user shows signs of confusion and asks the examiner to help him with what to do next, the system fails to meet this goal.

Usability Test Procedure

How many examiners are required?

We recommend using a minimum 2 examiners to manage each user test. One of the examiners should be there to help the user in case of question or confusion and the other should be recording if the usability goals above are being met. Ideally, 5 users should be tested,as depicted by the “Why You Only Need To Test With 5 Users” class reading.

What equipment will the examiners need?

The hardware equipment required for the testing is an android smartphone with a cover that can be worn around the neck so the phone can hang, and a pair of headphones, ideally bone conduction. If there is a problem finding bone conduction headphones or cover for the smartphone, please contact Roger from the Intersect-Assist team. The application runs on Android which means the users will need to install the application from the Google Play Store.

How should your prototype be handled?

The user/examiner should follow the procedure depicted in the user manual documentation above for installation. After the application has been installed, the hardware should be worn by the user, as depicted in the manual.

Seeing as how this is a simulation, this computer prototype is designed to map specific intersections only. As a result, doing field tests is not feasible since the veering mechanism is not put in place, which can put the users in a potentially dangerous situation.

The mapping provided in this prototype is based on specific walking paths the user would follow. If the user changes orientations, the information provided won’t be accurate. The simulated veering component is put in place to provide the user with a sense of how the application would portray the information regarding which way the person should veer.

DISCLAIMER: No actual sensing is being performed by the application. Do not let the user cross the intersection without your help.
How will you instruct your examiners to proceed?

The examiners should start by downloading the application and reading the user manual above. Afterwards, they should start the application and perform a cognitive walkthrough to find usability issues and potential redesign points. The examiners should follow the procedure depicted in the ‘Usability Evaluation’ section of this page.

Afterwards, the examiners should contact some potential users, as indicated in the ‘User Population’ section, and attempt to meet with them to perform the usability test. As mentioned in the previous point, we recommend performing the usability testing in a usability lab environment since actual intersection will not provide the user with the correct data and the outdoor environment might overload the user. Also, if the user chooses a route that was not predicted in this simulation, the user might get confused by the information being provided and, as a result, the usability goals may be affected.

What should the examiners tell the test subjects?

The examiners should debrief the subjects according to the ‘User Introduction’ section in the user manual. Afterwards, the examiners should proceed to the ‘Pretest Questionnaire’. It may be necessary to help the user install the application depending on how ‘Smartphone Savvy’ the user is. Once the test is complete, the examiners should proceed to the ‘Post-test Questionnaire’.

What should the examiners avoid doing?

The examiners should avoid providing the user with help unless the user asks for it. The user should be able to start the application by themselves, once installed on their smartphone.

The examiners should also avoid talking to the user when they are using the application. The user should be allowed to focus on the application since he/she is still in the novice phase.

Finally, the examiners should avoid telling or insinuating what the usability goals are. This can greatly influence how the user attempts to accomplish certain tasks in the application.

How and when should the examiners measure?

The examiners should be measuring according to the ‘Usability Goals’ document. This should be done throughout the usability test. If any goal, depicted in the Usability goals section, fails, it should be noted in this document. One document should be printed and filled out per test subject.

Usability Evaluation

The usability evaluation should be performed by each member of the team separately. It should be performed by filling the complete document including the comments section. Afterwards, we recommend the team meet together and further discuss the usability problems discovered during the individual sessions. Once the discussions are complete, the team should fill a paragraph describing the agreed upon usability problems discovered as a group. We would like to collect these documents from the evaluators once they complete the formative feedback section of the project.
